# Dastbadast Multivendor Delivery Ecosystem 🚴‍♂️🥘

**Dastbadast** (в переводе со ср.-перс. / тадж. _«Из рук в руки»_) — это комплексная программная экосистема автоматизации, заказа и доставки еды, разработанная специально для рынка Таджикистана (первоначальный запуск — г. Душанбе).

Проект спроектирован по архитектуре **монорепозитория** и включает в себя центральное GraphQL API ядро и 4 специализированных приложения (клиентское, ресторанное, курьерское и административное). Интерфейсы всей системы объединены единым дизайн-кодом в **тёмном POS-стиле** (Dark POS UI), оптимизированном для высокой контрастности и скорости работы.

## 🏗 Архитектура Монорепозитория и Технологический Стек

Репозиторий разделен на 5 автономных слоев (пакетов), каждый из которых решает свою бизнес-задачу:
dastbadast-multivendor/
├── dastbadast-multivendor-api/ # Центральный GraphQL API сервер
├── dastbadast-multivendor-web/ # Веб-сайт для конечных клиентов
├── dastbadast-multivendor-store/ # Планшетное приложение для ресторанов
├── dastbadast-multivendor-rider/ # Мобильное приложение для курьеров
└── dastbadast-multivendor-admin/ # Панель глобального управления (Суперадмин)

### 1. 🧬 API Сервер (`dastbadast-multivendor-api`)

- **Стек:** Node.js (ES Modules), Express, Apollo Server v4, WebSockets (`graphql-ws`), Mongoose / MongoDB.
- **Ключевой функционал:**
  - Единая GraphQL Schema, объединяющая Query, Mutation и Subscription.
  - Ролевая модель доступа (RBAC): `CLIENT`, `STORE`, `RIDER`, `ADMIN`, управляемая через JWT-токены в HTTP-заголовках (`Authorization: Bearer <token>`).
  - PubSub архитектура на базе веб-сокетов для мгновенной рассылки обновлений без частых HTTP-запросов (polling).
  - Интеграция с СУБД MongoDB: валидация полигонов (2dsphere индексы) для проверки зон доставки.

### 2. 🌐 Веб-клиент (`dastbadast-multivendor-web`)

- **Стек:** Next.js 14 (App Router), React, Apollo Client (SSR + Client-side-runtime), Tailwind CSS, Leaflet Maps, Lucide Icons.
- **Ключевой функционал:**
  - Интерактивная карта для выбора адреса доставки. Проверка координат пользователя на пересечение с полигоном зоны покрытия ресторана перед созданием корзины.
  - Трёхколоночный интерфейс на странице ресторана: Категории (левая) ➡️ Сетка блюд с модификаторами (центральная) ➡️ Липкая корзина (правая).
  - Динамический трекинг курьера на карте через GraphQL Subscriptions после перевода заказа в статус `PICKED_UP`.

### 3. 📱 Приложение Ресторана (`dastbadast-multivendor-store`)

- **Стек:** Expo v56 (Expo Router), React Native, TypeScript, Apollo Client.
- **Ключевой функционал:**
  - Интерфейс, адаптированный под планшеты, для работы на кассе заведения.
  - Фоновое прослушивание мутаций (`subscribePlaceOrder`). При поступлении заказа со статусом `PENDING` срабатывает визуальный алерт и циклический звуковой сигнал.
  - Три операционных экрана управления очередью: `New` (Новые) ➡️ `Processing` (Готовка) ➡️ `Ready/History` (Переданные курьеру / Архив).

### 4. 🛵 Приложение Курьера (`dastbadast-multivendor-rider`)

- **Стек:** Expo v56 (Expo Router), React Native, NativeWind (Tailwind для RN), `expo-location`.
- **Ключевой функционал:**
  - Экран `Pool` (Пул свободных заказов в Душанбе) с отображением тарифа, точек А (Ресторан) и Б (Клиент).
  - Цикл отслеживания геолокации (`Location.startLocationUpdatesAsync`): при взятии заказа запускается фоновый воркер, транслирующий GPS-координаты курьера через GraphQL-мутацию на сервер каждые 10-15 секунд.
  - Встроенный чат с рестораном и клиентом внутри карточки активного заказа.

### 5. 👑 Панель Администратора (`dastbadast-multivendor-admin`)

- **Стек:** Next.js 14, Tailwind CSS, Shadcn/ui референсы, ApexCharts.
- **Ключевой функционал:**
  - **Dispatch Монитор:** Живая интерактивная карта города с маркерами всех активных заказов, ресторанов и курьеров. Возможность принудительного ручного назначения курьера на заказ.
  - **Кабинет Accounting (Бухгалтерия):** Автоматический подсчет выручки заведений, начисление зарплаты курьерам по тарифной сетке и расчет чистой прибыли платформы (комиссия % от чека).
  - **Конфигуратор зон:** Рисование полигонов доставки на карте и привязка их к тарифам и ресторанам.

## 🎨 Спецификация Дизайн-Системы (Dark POS UI)

Все приложения используют строгую палитру цветов и правила верстки. Изменение цветов вопреки токенам системы запрещено для сохранения визуальной целостности бренда.

| Токен        | HEX / Значение | Применение в интерфейсе                          |
| :----------- | :------------- | :----------------------------------------------- |
| `dbd-bg`     | `#1F1D2B`      | Главный фон страниц, экранов и подложек          |
| `dbd-card`   | `#252836`      | Фон карточек товаров,サイドバー панелей, модалок |
| `dbd-accent` | `#EA7369`      | Кнопки действия, цены, бейджи, активные табы     |
| `dbd-border` | `#393C49`      | Тонкие разделители, границы полей ввода (Inputs) |
| `dbd-text`   | `#FFFFFF`      | Основные заголовки, читаемый текст, цифры чека   |
| `dbd-muted`  | `#92929D`      | Второстепенная информация, даты, плейсхолдеры    |

### Важные правила верстки:

- **Скругления:** Кнопки и инпуты — `rounded-xl` (12px). Карточки блюд и ресторанов — `rounded-2xl` (16px). Большие баннеры и модальные окна — `rounded-3xl` (24px).
- **Эргономика клика:** На мобильных устройствах (`rider`, `store`) высота любых кликабельных элементов и кнопок изменения статуса должна быть **не менее 48px** (оптимально `52px`) для минимизации ложных нажатий.

## 📂 Подробная Структура Базы Данных (Схема MongoDB)

Основные сущности и их связи, заложенные в ядре `dastbadast-multivendor-api/src/models/`:

1. **`User`**: Хранит данные пользователей всех ролей. Поля: `name`, `email`, `password` (hash), `phone`, `role` (`CLIENT`/`STORE`/`RIDER`/`ADMIN`), `isAvailable` (для курьеров).
2. **`Zone`**: Географические полигоны доставки. Поля: `title`, `location` (`type: "Polygon"`, `coordinates`), `tax` (базовая стоимость доставки в сомони для этой зоны).
3. **`Restaurant`**: Профили заведений. Поля: `name`, `slug`, `image`, `address`, `location` (`type: "Point"`), `zone` (`ObjectId -> Zone`), `owner` (`ObjectId -> User`).
4. **`Category` & `MenuItem`**: Меню ресторанов. Поля товара: `title`, `description`, `price` (в TJS), `image`, `restaurant` (`ObjectId`), `options` (массив модификаторов, например, соусы/размер).
5. **`Order`**: Сквозная сущность заказа. Поля:
   - `client`, `restaurant`, `rider` (`ObjectId`)
   - `items`: массив `{ menuItem, quantity, selectedOptions }`
   - `totals`: `{ subTotal, deliveryFee, total }`
   - `status`: `PENDING` ➡️ `ACCEPTED` ➡️ `PREPARING` ➡️ `READY` ➡️ `PICKED_UP` ➡️ `DELIVERED` / `CANCELLED`
   - `paymentMethod`: `COD` (Cash on Delivery / Наличные), `ALIF`, `DC`.
   - `deliveryAddress`: `{ title, lat, lng }`.

## 🔄 Жизненный цикл заказа (E2E-сценарий работы системы)

Экосистема работает как единый распределенный автомат состояний в реальном времени:

[Клиент: Web/App] --------(placeOrder Mutation)-------> [API Сервер]
|
(Subscription Broadcast)
|
v
[Курьер: Изменяет статус] <-- (Subscription) --- [Ресторан: Экран "New"]
| |
(status: PICKED_UP) (status: ACCEPTED)
| |
v v
[API: Запуск GPS Трекинга] [Кухня: Готовка еды]
|
v
[Клиент: Видит курьера на карте] ---> (status: DELIVERED) ---> [Admin: Accounting перерасчет]

1. **Размещение:** Клиент на сайте собирает корзину, выбирает тип оплаты `COD` и нажимает «Оформить». Срабатывает мутация `placeOrder`. Заказ сохраняется со статусом `PENDING`.
2. **Оповещение ресторана:** Через открытый WebSocket-канал срабатывает GraphQL Subscription `orderPlaced`. Приложение ресторана (`dastbadast-multivendor-store`) ловит событие, включает сирену и выводит заказ на экран `New`.
3. **Принятие и готовка:** Менеджер ресторана нажимает «Принять». Статус меняется на `ACCEPTED` (сирена выключается, заказ переходит в колонку `Processing`). Повар начинает готовить. По окончании менеджер жмет «Готово», статус становится `READY`.
4. **Назначение курьера:** В панели `dastbadast-multivendor-admin` в диспетчерской зоне заказ подсвечивается как готовый. Система находит ближайшего курьера со статусом `isAvailable: true` или админ назначает его вручную (`assignRider`).
5. **Доставка:** Курьер в приложении `dastbadast-multivendor-rider` получает пуш-уведомление, принимает заказ, приезжает в ресторан и нажимает «Забрал заказ» (`PICKED_UP`).
6. **Живой трекинг:** В момент статуса `PICKED_UP` в приложении курьера активируется фоновый цикл `expo-location`. Он шлет мутации `updateRiderLocation`. Веб-сайт клиента подписывается на изменения координат и плавно двигает маркер мопеда по карте Душанбе.
7. **Финал:** Курьер отдает пакет клиенту, нажимает «Доставлено» (`DELIVERED`). Статус закрывается. Административная панель мгновенно обновляет графики выручки, прибавляет курьеру баланс за выполненную работу, а ресторану начисляет выплату за вычетом комиссии платформы.

## 🛠 Руководство по развертыванию локального окружения

### Шаг 1: Клонирование и Docker

Склонируйте репозиторий, перейдите в корень проекта и запустите контейнер базы данных MongoDB:

```bash
docker compose up -d mongo
Шаг 2: Настройка и запуск API сервера
Bash
cd dastbadast-multivendor-api
cp .env.example .env
npm install

# Запуск сид-скрипта для наполнения БД (создаст зоны Душанбе, рестораны, меню и аккаунты)
npm run seed

# Запуск сервера в режиме разработки
npm run dev
# API будет доступно по адресу: http://localhost:8001/graphql
Шаг 3: Запуск Веб-платформы (Клиентская часть)
Bash
cd ../dastbadast-multivendor-web
cp .env.local.example .env.local
npm install
npm run dev
# Сайт запустится на http://localhost:3000
Шаг 4: Запуск Панели Администратора
Bash
cd ../dastbadast-multivendor-admin
cp .env.local.example .env.local
npm install
npm run dev
# Панель запустится на http://localhost:3001
Шаг 5: Запуск Мобильных приложений (Expo)
Для запуска приложений на симуляторах (iOS/Android) или через физическое устройство с приложением Expo Go:

Bash
# Для запуска терминала Ресторана
cd ../dastbadast-multivendor-store
npm install
npx expo start

# Для запуска приложения Курьера
cd ../dastbadast-multivendor-rider
npm install
npx expo start
Важно: Если вы тестируете приложения на реальном смартфоне через Expo Go, замените localhost на локальный IP-адрес вашего компьютера (например, 119.168.1.X) в файлах конфигурации .env мобильных приложений, иначе они не смогут достучаться до API.

🔑 Учетные записи для тестирования (Тестовый стенд)
```
